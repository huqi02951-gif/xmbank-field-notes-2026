# 《实务研习院》体验升级设计规范

## 一、产品定位

本项目不是宣传页、低龄化入职指南或任务打卡系统，而是一份面向台湾高校实习生的高密度实务课程手册。它借鉴顶尖高校课程大纲的学术严谨、专业服务机构的训练方法和编辑刊物的阅读体验，帮助实习生在一个月内完成四次能力跃迁：从观察现象，到提出好问题；从整理材料，到形成判断；从接受指导，到独立表达。

网页首先服务实习生的学习与安全，其次才承担导师形象表达。导师的专业、责任感与创造力不通过自我评价呈现，而通过清晰课程设计、准确边界、可复核工具、及时反馈机制和对语言差异的照顾自然被感知。

## 二、受众、语气与内容原则

核心受众是第一次进入大陆银行实务环境的台湾高校学生。他们具备大学阶段的理解力与独立判断能力，但可能不熟悉大陆银行术语、简体书写、组织流程和对客边界。

内容采用正式、克制、友善的导师口吻：不训诫，不讨好，不制造虚假的轻松感，不把复杂业务简化成口号。每个章节都应提供值得反复阅读的问题、方法或工具。专业陈述必须能够追溯、标注适用边界，并明确区分：

1. 可用于对客沟通的已核准表达；
2. 仅用于理解业务结构的教学示例；
3. 需要向导师、业务负责人或当地网点复核的动态信息；
4. 不得收集、复制、上传或公开的客户及内部信息。

简体是唯一内容源，繁体由受保护术语转换层生成。阅读、APEX 输出和复制均可分别选择简体或繁体，使台湾实习生能够先用熟悉文字理解，再获得适合大陆工作场景的简体表达。

## 三、信息架构

### 1. Opening Note / 开始之前

用一封短而有分量的导师序言建立心理契约：允许暂时不知道答案，但要求认真观察、核对事实、尊重边界，并形成自己的判断。首屏同时给出课程周期、四项学习成果和主行动入口。

### 2. Curriculum / 一个月的能力路径

以四个阶段组织学习，而不是以待办事项组织管理：

- Observe：进入现场，识别角色、流程、材料与语言；
- Question：从现象提出能够被验证的问题；
- Synthesize：把零散信息整理为结构、风险与下一步；
- Communicate：用准确、适当、可复核的方式对客或汇报。

每一阶段包含核心问题、建议实践、可见产出和导师反馈点。进度由读者自主展开，不做积分、排行榜或幼稚化徽章。

### 3. Seminar / 银行业务研讨

重构现有“银行里值得看的事”和三个业务瞬间，使其成为三组案例研讨。每组以真实问题开场，依次呈现观察、证据、判断边界与讨论题。内容聚焦企业开户、企业轮廓、授信问题等结构性理解，不提供未经复核的产品承诺、费用、办理时效或内部规则。

### 4. APEX Lab / 可复核工具实验室

保留现有企业开户材料助手，并将其从展示组件提升为课程实验：选择开户场景和临时材料后，输出已准备、待补充和复核声明。工具只使用脱敏公开教学数据，零网络请求、零业务数据存储、零客户输入。

实验室明确展示“输入—专业能力—可复核结果”的工作方式，并说明工具不能替代网点最新要求。复制操作支持当前文字、简体和繁体三个选择。

### 5. Practice / 第一次进入现场

以四个高价值时刻组织行动建议：第一次进入部门、第一次走访客户、第一次接触材料、第一次参与复盘。每个时刻提供“观察什么、记录什么、不要做什么、回来后问什么”，同时强调客户隐私、保密、授权和对客表达边界。

### 6. Research Studio / 课题工作室

将导师原件中的个人与团队课题转化为通用研究方法：定义问题、获得证据、分析结构、制作原型、形成表达。提供优秀成果的质量标准：问题真实、证据可解释、建议可实施、原型可体验、边界被说明。网页不复制原件中的个人信息或内部提交渠道。

### 7. Mentorship / 我们怎样一起工作

以协作协议而非导师宣言呈现：短周期复盘、问题清单、课题检查点、分层工作安排、鼓励式反馈、公平互评和异常情况及时上报。实习生能够知道何时应自行探索、何时应请求确认、何时必须立即上报。

### 8. Field Notes / 下班以后

保留城市生活入口，但把目的从消费推荐调整为恢复精力、理解城市和记录共同经历。只提供可靠的选择方法与安全提醒，不编造店名、评分、价格或路线。

### 9. Closing Reflection / 带走什么

以一组开放问题结束：我现在能看见什么、能提出什么问题、能整理什么证据、能向他人解释什么。允许复制一份私人复盘模板，但不在网页保存任何回答。

## 四、视觉与交互方向

视觉主题为“海港研究院档案”：融合专业机构研究报告、大学课程手册和私人批注本。主色使用深海军蓝与温暖纸色，辅助色为海水蓝、档案红和少量黄铜色。排版建立明显的学术层级：大字号编辑标题、紧凑的元数据、舒展的正文、等宽工具输出和克制的边注。

页面采用非对称十二列网格，章节编号、课程标签、引文和批注形成可识别的阅读节奏。不同内容使用不同构图，不批量套用圆角卡片。动效只用于首屏路线、章节进入、课程阶段切换和实验结果更新，并完整支持减少动效。

交互必须满足键盘操作、可见焦点、语义化标题、折叠状态提示、移动端无横向溢出和简繁切换不改变滚动位置。页面在 375、390、430、768 和 1440 像素宽度下均应保持可读。

## 五、技术架构与数据边界

继续使用 Vite、React、TypeScript 和结构化原生 CSS，输出可静态托管的 `dist/`。不引入后端、登录、数据库、分析脚本或客户数据表单。现有 APEX 适配层保持只读，只有经脱敏、可追溯的类型、教学数据和纯函数进入公开站点。

内容模块与视觉组件分离，单一简体内容源承载全部正式文案。页面状态只包含语言、展开项和 APEX 临时选择；除语言偏好外不持久化任何数据。公开构建不得包含 PDF、个人姓名、联系电话、住宿地址、本地路径、密钥、内部网址或原始 APEX 配置。

## 六、验证与发布

发布门槛包括 ESLint、TypeScript、Vitest、内容安全审计、Playwright 桌面与移动端测试、关键无障碍检查和 Vite 生产构建。需要验证简繁转换、复制语言、APEX 临时状态、内部锚点、键盘流程、移动端布局、控制台错误和静态子路径资源。

首选 GitHub Pages：公开仓库通过 GitHub Actions 执行审计、测试与构建，再发布 `dist/`。若当前 GitHub 权限、仓库可见性或 Pages 策略阻止部署，使用等价的静态托管服务，并保留相同的构建产物和验证门槛。最终交付一个可直接打开的 HTTPS 地址、构建与测试结果以及后续内容维护说明。

## 七、非目标

- 不制作宣传片式企业官网；
- 不制作积分、打卡、排行榜或幼稚化课程系统；
- 不虚构客户案例、生活地点、评分或业务数据；
- 不把教学示例包装成实时业务规则；
- 不收集实习生、客户或企业信息；
- 不迁移 APEX 后端、密钥、内部风控规则或持久化能力；
- 不为了视觉效果牺牲阅读、性能、准确性与无障碍。

## 八、完成标准

成品应让实习生在首次浏览后明确知道：这一个月为什么值得投入、应该如何学习、怎样把观察变成能力、哪些内容可以对客使用、哪些必须复核、导师会如何协作，以及遇到风险时应怎样处理。页面应具有再次阅读的价值，并通过结构与细节，而非自我宣传，使读者感受到导师准备充分、技术可靠、判断成熟且尊重学习者。
