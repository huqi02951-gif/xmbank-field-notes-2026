# 《两岸实习手记》UI 与体验设计审计及规格草案

> 文档性质：开发前设计基线，供内容、前端、测试与截图验收共用。  
> 适用范围：单页静态站《两岸实习手记》，主要视口为 390 × 844 px。  
> 当前证据：已确认产品需求与工作区现状；工作区尚无可运行页面、视觉稿或现有截图，因此本文是“前置设计审计 + 可执行规格”，不是对已实现界面的视觉验收。

## 1. 审计结论

整站应被设计成一本“认真编辑的私人数字手册”，而不是课程网站、行内宣传页或 AI 产品展示页。体验成功的核心不在于规模，而在于三个感受同时成立：

1. **低压力**：用户随时可以停止，不被进度、作业、打卡或“必须学会”的语气绑架。
2. **真专业**：不回避银行业务的层次，但每一段只帮助用户理解一个问题。
3. **有人情味**：视觉像一位懂业务、也懂生活的导师在交付一份手记，不像统一模板生成的页面。

首屏先降低心理门槛，中段用真实业务瞬间建立理解，APEX 区用“复杂被整理清楚”建立能力信任，末页再以轻生活入口收板。不建议在首屏讲银行历史，也不建议用四个并列功能卡介绍 APEX。

## 2. 用户心理与体验原则

### 2.1 用户当下的心理

| 心理状态 | 隐性问题 | 页面回应 |
| --- | --- | --- |
| 初入陌生银行环境，担心听不懂 | “我会不会连基本概念都不知道？” | 从三个真问题进入，先给理解框架，不先考术语。 |
| 担心实习再变成一套课程 | “我是不是又要打卡、交作业？” | 首屏直说“不用按顺序阅读”；全站不出现进度或完成状态。 |
| 想看学校里看不到的东西 | “真实业务到底怎么被判断？” | 用对话、材料台、四问三种不同叙事结构展示判断过程。 |
| 希望未来面试能说出收获 | “我能带走什么可复述的经验？” | 每个业务瞬间收束为一个可记住的判断句。 |
| 想知道导师是否好沟通 | “我有问题敢不敢问？” | 文案用“可以试着”“随时问”，少用“应当”“必须”。 |
| 也关心下班后的自由时间 | “这里是不是只有工作？” | 末页给三个轻入口，但不代替用户做口味和路线决定。 |

### 2.2 体验准则

- **先回答“为什么值得看”，再给专业细节。**
- **一屏一个主角。** 一个问题、一组材料或一个结果，不并置多个争抢注意力的卡片。
- **展开是选择，不是任务。** 折叠内容最多一屏可读完，不引入下一步压力。
- **结论可被带走。** 重要段落的末句应能单独成立，便于截图、转发或面试时复述。
- **不做身份猜测。** 语言选择、饮食和生活偏好都交给用户决定。
- **专业信任来自过程透明。** APEX 不拟人化，不伪装实时 AI，不展示无法核实的效率数字。

## 3. 视觉方向

### 3.1 设计命题：海风里的业务手记

主体是米白纸张、深蓝文字、克制的厦门蓝和浅海玻璃色；橙、黄、咖啡色只在日期章、票根、便签和生活区中出现。黑体系统字体、明显左对齐和大面积留白形成专业底色；手绘线、编号、边注和纸质细节则提供个人感。

参考的是国际交流项目页的清晰层级、律所与 Early Careers 页面的克制、文化杂志的编辑节奏；不复制其品牌、布局、字体或组件。

### 3.2 视觉节奏

1. **首屏：空气感。** 约 65% 的安静空间，标题与细路线对话。
2. **银行问题：清晰感。** 纵向分隔、少量边注，不做卡片墙。
3. **业务瞬间：变化感。** 对话、工作台、大号数字三种版式连续出现，每段都换一种阅读动作。
4. **APEX：秩序感。** 从左侧无规则的材料碎片进入细窄处理轨迹，右侧输出对齐的结果。
5. **生活区：松弛感。** 暖色稍增，版式像杂志末页，但不变成旅游攻略。

### 3.3 图像与插画

- 首屏的海岸线和城市轮廓应是真实授权的主视觉资产，不用线上地图截图，不用政治地图，不用桥梁连接两岸的俗套隐喻。
- 若没有已授权的照片，生活区宁可保持纯排版，也不使用不明来源图片。
- 禁用金币、钱袋、人民币符号、机器人、发光大脑、粒子流、广告模特式握手照。

## 4. 信息架构与移动优先低保真结构

### 4.1 整体流程

```text
固定顶栏：项目名                                      简体 | 繁體
│
├─ 01 首屏：两岸实习手记 + 路线主视觉 + 从这里开始
│
├─ 02 银行里值得看的事：三个可展开问题 + 轻量年份边注
│
├─ 03 三个真实业务瞬间
│     ├─ 对话式：开户前为什么要了解企业
│     ├─ 材料工作台：零散信息到企业轮廓
│     └─ 四问式：谁 / 为什么 / 靠什么 / 变化时怎么办
│
├─ 04 窄纸条：“留下一句话” + 复制
│
├─ 05 APEX 精选工具：碎片→处理→结果 + 可安全迁移的轻量工具/示例
│
├─ 06 下班以后：徒步 / 咖啡甜点奶茶 / 吃什么
│
└─ 极简页脚：科技业务部 XD.H / 仅供个人交流
```

### 4.2 390 px 首屏低保真结构

```text
┌──────────────────────────────┐
│ 手记                         简体 繁體 │  56 px 固定顶栏
├──────────────────────────────┤
│ 2026 · 厦门银行                         │
│                                              │
│ 两岸实习                                 │
│ 手记                                      │  标题两行
│                                              │
│ 欢迎来到厦门。                            │
│ 这里整理了一些银行里值得看的事…          │  文案 4–5 行
│                                              │
│ [ 从这里开始 ]                              │  44 px 以上
│                                              │
│        • 银行 ─── • 企业                   │
│       ／                      • 城市          │  主视觉
│  ≋≋ 抽象海岸线 / 城市轮廓 ≋≋                 │
└──────────────────────────────┘  总高 ≤ 95vh
```

要点：首屏不强行放完全部长文案；文案与主视觉各自有空间。语言切换保持可见，导航如需收起，也不得遮住语言切换。

### 4.3 银行问题与业务瞬间

```text
银行里值得看的事

01  为什么企业开户比个人开户复杂？       [+]
───────────────────────────────
02  为什么客户说经营很好，银行仍会继续问？ [+]
───────────────────────────────
03  为什么简单业务需要这么多人确认？   [+]

                     1996  厦门银行成立
                     2013  青年实习合作启动
                     2026  这一届实习交流

── 业务瞬间 01 / 对话──
客户话语（淡底色）
      银行关注（四行编辑式列表）
一句可记住的总结

── 业务瞬间 02 / 材料台──
材料碎片（不超过六项，大小不同）
       ↓ 整理
六个判断问题（对齐列表）

── 业务瞬间 03 / 四问──
01 谁在申请？
02 为什么需要？
03 靠什么偿还？
04 变化时怎么办？
```

问题展开后，加减号或箭头必须表达当前状态；标题整行可点，点击区高度不小于 56 px。年份只是边注，不做横向大时间轴。

### 4.4 APEX 区移动低保真结构

```text
经验怎样被整理下来
导语（4–6 行）

┌ 原始信息 ──────────────────┐
│ 企业基本信息      客户描述             │
│   缺失材料        不一致信息           │
│           专业问题                       │
└────────────────────────────┘
                     ↓
             APEX
   识别任务 → 调用专业能力 → 形成可复核结果
                     ↓
┌ 清晰结果 ────────────────────┐
│ 开户材料清单                              │
│ 一页企业画像                              │
│ 对客补充说明                              │
│                                             │
│ [复制当前版本]                         [···] │
└────────────────────────────┘
```

不使用三张等宽卡片。输入碎片边缘可略有错落，处理轨迹窄而安静，结果区回到严格对齐。右侧工具若不能安全迁移，必须明标“示例”，不伪装实时智能调用。

### 4.5 下班以后

移动端先放一张经授权照片（若无则整块不渲染），再放三行可展开入口。展开内容只说如何开始寻找，不展示空推荐卡、未验证店名、排行榜或预设口味。

## 5. 两个截图记忆点

### 5.1 记忆点 A：“一本专门准备的手记”

**截图尺寸**：首先优化 390 × 844 px，同步验收 1440 × 1000 px。

**画面主角**：“两岸实习手记”标题和一条从纸面游走的细路线。三个节点只写“银行 / 企业 / 城市”。左上角是小号“2026 · 厦门银行”，右上角保持语言选择。

**记忆原因**：不用企业宣传照也能形成地域与个人感；文案明确取消课程压力，使用户立刻理解这是可自由翻阅的手册。

**不得破坏画面的内容**：全屏高饱和蓝、人物抓拍、政治地图、桥梁隐喻、导航项过多、标题超过两行、超大英文衬线副标。

### 5.2 记忆点 B：“复杂被整理清楚”

**截图尺寸**：单独验收 390 × 844 px 的 APEX 区完整一屏；桌面验收 1440 × 1000 px。

**桌面画面**：左侧是不同大小的材料碎片，中间是很窄的 APEX 处理轨迹，右侧是对齐、可复制的三个结果。左乱、中净、右稳，空间本身表达能力。

**移动画面**：严格纵向，但不是四个一模一样的蓝色流程框。输入保留材料感，中段 APEX 是短暂、有呼吸的过渡，结果区以按钮和清晰行距收束。

**记忆原因**：它不用 AI 视觉俗套，直接展示“信息如何变得可复核”，比聊天气泡和抽象架构图更能传达 APEX 的真实价值。

## 6. Design Tokens

下列 token 是视觉基线；前端应在 `src/styles/tokens.css` 中统一实现，不在组件里重复写色值。

| --- | --- | --- |

| `--deep-harbour` | `#173D56` | 深色标题、APEX 秩序区 |
| `--sea-glass` | `#DCECF0` | 淡区分、问题展开底色 |
| `--brick` | `#B96049` | 日期章、重点批注 |

- 桌面 12 列，手机 4 列；不把桌面双栏直接压缩到手机。
- 阴影只用于必要浮层和经授权图片；普通内容块依靠留白、分隔线和底色建立层级。
- 圆角定为功能性细节，不是全局风格；按钮、结果面板和便签可不同形。

### 6.4 动效

| 类型 | 时长 | 原则 |
| --- | --- | --- |
| 点击/切换 | 150–250 ms | 直接，无弹簧式夸张反馈 |
| 区块进入 | 350–500 ms | 淡入 + 最多 12 px 上移 |
| 首屏路线 | 800–1200 ms | 仅播放一次，不循环 |
| APEX 示例 | 150–250 ms | 结果替换保持容器稳定，避免布局跳动 |

`prefers-reduced-motion: reduce` 下取消路线绘制和区块位移，保留必要的瞬时状态切换。

## 7. 组件视觉与交互规则

| 组件 | 视觉角色 | 关键规则 | 禁止项 |
| --- | --- | --- | --- |
| `Header` | 安静的阅读辅助 | 高 56–64 px，可半透明纸色；移动端始终保留语言切换 | 超过 5 个主导航项、大面积蓝底 |
| `LanguageToggle` | 两态文字开关 | 同宽选项，当前态用底线/实色文字，按钮高度 ≥44 px；切换不改变滚动位置 | 国旗图标、同屏双语全文 |
| `Hero` | 整站的编辑封面 | 标题最多两行；一个主按钮；路线与标题平衡；总高 ≤95vh | 双主按钮、超大英文、整屏高饱和色 |
| `BankQuestions` | 无边框纵向索引 | 每行用细分隔线；整行可点；展开文案最多一小段；使用语义化折叠状态 | 三张等宽卡片、图标墙 |
| `ScenarioDialogue` | 编辑式对话 | 客户原话是淡底色引文，银行关注是左对齐条目，总结单独收束 | 聊天气泡、人物头像、“AI 对话”外观 |
| `ScenarioWorkspace` | 材料整理桌面 | 碎片不超过 6 项，尺寸和排列有节奏；整理后六问回归网格 | 仿真客户文件、真实证照缩略图 |
| `RiskQuestions` | 大号四问 | 01–04 是导视主体，留白大于装饰；最后一句像页边批注 | 风险评分、完成勾选、进度环 |
| `NoteStrip` | 窄纸条/票据 | 宽度不占满大屏，三行句式不可编辑，一个复制主操作 | 输入框、保存状态、每日提醒 |
| `ApexWorkbench` | 专业工作台 | 桌面左—中—右，移动上—中—下；只说识别、调用、复核 | 四个并列蓝框、发光轨迹、复杂架构图 |
| `ApexTool` | 输出区中的小工具 | 输入少且来自已核实逻辑；结果有“当前版本”语言状态；刷新即清空 | 注册、存储、上传、聊天框、伪实时 AI |
| `CopyButton` | 主操作 + 次级菜单 | 主按钮复制当前语言；次级菜单明确提供简体/繁体；键盘可达 | 只用图标无文字名称 |
| `Toast` | 短暂确认 | 不挡住底部主操作；约 2–3 秒；文案说明已复制哪种语言 | 长时间驻留、只靠颜色区分 |
| `LifeCorner` | 杂志末页 | 暖色增加；照片如有则在移动端置顶；三个入口展开后仍是短文 | 店铺排行、空卡、身份导向推荐 |
| `Footer` | 极简署名 | 只显示“科技业务部 XD.H”与“仅供个人交流” | 友链、社交媒体图标、营销口号 |

## 8. 可访问性与响应式验收基线

- 正文与背景、按钮文字与按钮底色需经实测达到 WCAG 2.2 AA；本规格不单凭色值声称已合规。
- 主要标题层级从 `h1` 到 `h2/h3` 连续，不为视觉尺寸跳级。
- 所有折叠、语言切换、复制菜单可用键盘完成；焦点态不可被清除。
- 交互目标最小 44 × 44 px；问题列表建议整行高度不小于 56 px。
- 折叠状态、当前语言、复制结果不仅靠颜色表达。
- 首屏路线是装饰时对辅助技术隐藏；若它承载“银行 / 企业 / 城市”信息，同样文本需以可读结构提供。
- Toast 应以非侵入式状态通知辅助技术，但不应抢占焦点。
- 200% 缩放和 320 px 等效宽度时内容仍可线性阅读；无横向滚动。
- 需分别截图验收 375、390、430、768、1440 px；不能只看桌面端。

## 9. AI 模板感风险清单

下列每项在视觉验收时均应以“不存在”为通过条件。

### 9.1 结构风险

- [ ] 每个区块都是“小标签 + 大标题 + 一段说明 + 三张卡片”。
- [ ] 三个业务瞬间使用同一版式。
- [ ] 银行问题、APEX 输入、APEX 输出和生活入口都被做成相同圆角卡。
- [ ] 每屏都居中对齐，缺乏编辑节奏。
- [ ] 每个区块都有一个“了解更多”按钮。

### 9.2 视觉风险

- [ ] 使用蓝紫渐变、发光球体、玻璃拟态或粒子。
- [ ] 使用超大英文衬线标题假装编辑感。
- [ ] 所有卡片均带阴影，所有按钮均为胶囊形。
- [ ] 高饱和厦门蓝占据整屏。
- [ ] 图标数量超过实际交互所需，或用 emoji 充当视觉系统。
- [ ] 将金币、钱袋、人民币符号当作银行视觉代表。

### 9.3 内容与 AI 表达风险

- [ ] 出现“赋能”“智慧引擎”“开启新篇章”等空洞口号。
- [ ] APEX 被表现为聊天机器人或会自主决策的“大脑”。
- [ ] 展示未经核实的效率百分比、自动决策或风险判断承诺。
- [ ] 为完整版式自动生成餐厅、徒步路线、评分、价格或未核实照片。
- [ ] 对台湾学生的饮食、文化或软件习惯做身份化推断。

### 9.4 动效风险

- [ ] 文字逐字播放，元素持续漂浮或鼠标尾迹。
- [ ] 每个组件都淡入、缩放或翻转。
- [ ] 滚动劫持、大幅视差、3D 翻转或自动播放媒体。
- [ ] 未支持 `prefers-reduced-motion`。

## 10. 开发后设计验收清单

1. **首屏截图**：390 × 844 内标题不超过两行，主按钮与主视觉完整可见，页高不超过 95vh。
2. **APEX 截图**：一眼可读出“零散信息 → APEX → 清晰结果”，不是等宽蓝色流程框。
3. **结构差异**：对话、材料台、四问、纸条、APEX 工作台、生活末页六种版式可明显区分。
4. **语言状态**：简体/繁体切换后保留滚动位置，且当前语言状态可见。
5. **空内容**：生活数据为空时不出现空卡和“待补充”。
6. **内容压力**：全页扫描不存在作业、截止时间、进度、打卡、必读和评分语言。
7. **手机重排**：375、390、430 px 无横向滚动，APEX 和图文双栏均为真正纵向重排。
8. **键盘与焦点**：按 Tab 顺序与阅读顺序一致，可看到焦点，菜单展开后焦点不丢失。
9. **动效限额**：只存在首屏路线、区块进入、APEX 示例切换、生活入口展开四类动效。
10. **安全与真实性**：页面无未授权人物照片、真客户材料、伪造店铺/路线和伪实时 AI 表达。

## 11. 证据限制与后续风险

- 当前没有可运行站点，未能执行实际流程截图审计；本文的可访问性内容是实现和验收目标，不是合规证明。
- 首屏主视觉尚无获准资产；若后续需新建图像，应单独生成并做版权、缩放和手机裁切验收，不以临时 CSS/SVG 拼凑代替主视觉资产。
- APEX 可操作工具的真实程度取决于只读审计结果；在来源逻辑、公开安全性和前端可运行性未确认前，不应为赶设计进度伪造一套“智能”输出。
- 简繁转换可能改变文字长度；需以 `zh-CN` 与 `zh-TW` 两态分别进行 390 px 截图验收，尤其检查标题两行限制、按钮宽度与 APEX 结果换行。
- 现有工作区只有私有源材料，无视觉实现；下一次审计应在首个可运行版本后，依次捕获首屏、问题展开、三个业务瞬间、APEX 结果和生活入口五类状态。
